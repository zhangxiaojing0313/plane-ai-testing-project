6. LOCATOR STRATEGY
================================================================================

Priority order (all based on real page exploration):
  1. getByRole (button, heading, link with accessible names)
  2. getByPlaceholder (password input)
  3. input[name=email], input[type=password] (precise attribute selectors)
  4. getByText for exact text matching
  5. locator().filter({ hasText: ... }) for contextual matching

Key locators discovered:
  - Email input:     input[name=email] (unique to login page)
  - Continue btn:    getByRole("button", { name: "Continue" })
  - Password input:  input[type=password] (unique to password step)
  - Login btn:       getByRole("button", { name: /go to workspace/i })
  - Workspace:       getByRole("heading", { name: "QA Lab" })
  - Project:         getByRole("link", { name: /Plane QA Project/i })
  - Work items nav:  getByRole("link", { name: "Work items" })
  - Add work item:   locator("button").filter({ hasText: /Add work item/i })
  - Title input:     locator("input[name=name]")
  - Save btn:        getByRole("button", { name: "Save" })
  - Delete btn:      getByRole("button", { name: "Delete" })

8. AUTOMATION LIMITATIONS (NOT PLANE DEFECTS)
================================================================================

The following are automation-specific constraints, NOT Plane product defects:

  1. force:true on Delete button — sidebar DOM elements may intercept
     pointer events in the drawer overlay. This is a locator/layout
     interaction issue in the test harness, not a Plane bug.

  2. waitForTimeout after SPA navigation — Plane renders client-side
     without explicit loading indicators. These waits are a pragmatic
     compromise for React hydration latency.

  3. Save button not disabled on empty title — Plane v1.3.1 defers
     validation to the server. This is documented behavior, not a defect.
