## 3 分钟版

### 项目背景

Plane 是一个开源的项目管理平台（类似 Jira 的开源替代）。我用五天时间为其搭建了一套完整的 AI 辅助自动化测试体系，从环境部署到 CI 集成全链路覆盖。

### 测试架构

采用经典的三层测试金字塔：

```
       ┌──────────┐
       │ UI (10)   │  Playwright + Chromium
       ├──────────┤   登录→工作区→项目→工作项 CRUD
       │ API (37)  │  pytest + requests
       ├──────────┤   认证/用户/工作项/项目/成员/状态/标签
       │ DB (19)   │  pytest + psycopg
       └──────────┘   Schema 验证 + API-DB 一致性断言
```

总计 **66 条唯一业务用例**，全部通过。

### CI 流水线设计

Jenkins Pipeline 包含 12 个阶段：

1. 代码同步（rsync from source mount）
2. 环境检查
3. Python 环境准备
4. Node 依赖安装
5. **UI 凭据冒烟** — 在执行全量测试前先验证凭据可用性
6. 创建临时 .env
7. API Validation (8 条)
8. API Regression (29 条)
9. 限流冷却 45s
10. Database Tests (19 条)
11. Playwright UI Tests (10 条)
12. 清理验证（自动删除测试数据，确认零残留）

### 技术决策

- **凭据管理**：所有敏感信息通过 Jenkins `withCredentials` 注入，代码中零硬编码。敏感操作区域 `set +x` 防止日志泄露。Post always 清理 `.env` 和 `auth-state.json`。
- **限流处理**：API 阶段后设 45s 冷却 + Python 端指数退避重试（2/4/8/16/32s）。
- **数据清理**：Cleanup Verification 阶段自动检测并删除所有 `AUTO-D` 前缀的测试数据。
- **Flaky 容忍**：Playwright CI 配置 `retries: 1`，UI-007 首次超时后自动重试通过。

### 排障亮点

**Jenkins Build #4–#8：凭据问题排查**

这是整个项目最有价值的排障案例：

1. 在 Agent 容器内手动执行 `npx playwright test auth.setup.ts` → 通过
2. 同一测试通过 Jenkins withCredentials 注入凭据 → 失败（回到邮箱输入步骤）
3. 结论：Playwright、页面对象、Plane 服务都正常，问题唯一指向 Jenkins 凭据
4. 在 Jenkins 中重建凭据记录（v2），更新 Jenkinsfile 绑定
5. Build #10 验证通过

这体现了 CI 排障的核心方法论：**逐层排除，定位最小故障域**。

### AI 协作模式

| 角色 | 工具 | 职责 |
|---|---|---|
| 测试架构师 | ChatGPT | 方案设计、问题分析、结果审核 |
| 代码工程师 | Codex | 测试代码、文档、代码审查 |
| 执行工程师 | Claude Code | Linux 执行、日志、部署、报告 |

最终所有判断由人工确认 — AI 负责加速，不替代决策。

## 面试官可能追问的问题与回答

### Q1: 为什么 API 是 37 条而不是 66 条？

**A:** 66 条是三层测试的唯一业务用例总和 — 37 条 API + 19 条数据库 + 10 条 UI。API Validation 的 8 条是环境/连通性验证，API Regression 的 29 条是业务功能回归。UI Credential Smoke 的 1 条 auth.setup 是 CI 前置校验，不计入唯一业务用例。这个分层统计方式让面试官能清楚看到每一层的覆盖深度。

### Q2: 你怎么判断 UI-007 是 flaky 而不是产品缺陷？

**A:** 三个判断依据：(1) retry #1 立即通过，说明不是系统性的功能异常；(2) 现象是"未在 10 秒内找到修改后的元素"，这是典型的异步加载时序问题而非功能缺失；(3) 在手动测试和多次 CI 运行中，该用例最终都通过。如果是产品缺陷，同样的操作无论重试多少次都应该失败。我的处置是标记为"自动化稳定性观察项"，不修改代码，后续采集 trace 确认根因后再优化等待策略。

### Q3: 为什么没有发现产品缺陷？

**A:** 这个项目的目标不是对 Plane 做深度缺陷挖掘，而是建立一套可复现、可审计的自动化测试骨架，验证核心业务流程在正常路径下可用。我们的用例设计偏向正向流程和关键异常路径，没有覆盖边缘场景的模糊测试或探索性测试。如果要发现缺陷，下一步应该增加边界值测试、权限越权测试、并发操作测试和长时间运行稳定性测试。

### Q4: 如果让你改进这个项目，最优先做什么？

**A:** 三个优先级：(1) **高** — 对 API 响应体做 JSON Schema 严格校验，当前只验证了状态码和关键字段，结构化断言能捕获更多回归；(2) **高** — 扩展 UI 测试覆盖权限边界和多用户协作场景；(3) **中** — 分析 UI-007 的 Playwright trace，精确定位根因并优化等待条件，消除 flaky。

### Q5: Jenkins 凭据问题给你什么启发？

**A:** CI 环境中最隐蔽的故障不是代码 bug，而是配置漂移。手动测试通过但 CI 失败时，最常见的三个原因：(1) 环境变量/凭据不一致；(2) 文件路径/权限差异；(3) 网络拓扑差异（如 localhost vs host.docker.internal）。排查这类问题的方法是逐层排除 — 先确认代码和框架正常，再确认被测服务正常，最后锁定配置层。这个思路在本项目的 Build #4-#8 排障中直接指导了凭据重建的解决方案。

### Q6: 你为什么把测试数据清理放在流水线里而不是用事务回滚？

**A:** Plane 的 API 是无状态的 HTTP 接口，不存在数据库事务的概念。测试数据通过 API 创建后持久化在 Plane 的 PostgreSQL 中，如果不清理会不断累积并影响后续测试（比如按名称查重）。我选择在流水线最后阶段做显式清理而不是依赖 tearDown，因为：(1) 如果某个测试中途崩溃，tearDown 可能不执行；(2) Post always 机制保证无论成功失败都会清理；(3) Cleanup Verification 阶段做二次确认，确保零残留。

## Jenkins 排障案例

### 场景：UI 登录在 Jenkins 中失败但手动通过

**背景**：Playwright UI 测试在 plane-qa-runner-agent 容器内手动执行通过（1 passed），但 Jenkins withCredentials 注入凭据后失败，浏览器返回到邮箱输入步骤。

**排障步骤**：

```
Step 1: 假设代码有问题
  → 手动执行 auth.setup.ts → PASS
  → 排除 Playwright、页面对象

Step 2: 假设 Plane 服务有问题
  → 手动输入账号密码登录 → 成功
  → 排除 Plane 服务

Step 3: 假设 Jenkins 凭据有问题
  → 对比手动输入的值与 Credential 记录
  → 确认凭据内容不一致

Step 4: 解决
  → 新建 plane-ui-email-v2 / plane-ui-password-v2
  → 更新 Jenkinsfile withCredentials 绑定
  → Build #10 验证通过
```

**方法论**：逐层排除，从代码→框架→服务→配置，定位最小故障域。

## API 限流案例

### 场景：API 测试批量返回 429

**现象**：API 连续调用时，部分接口返回 429 Too Many Requests。

**分析**：Plane API 存在速率限制机制，短时间内大量请求会触发限流。

**解决**：

1. **流水线层面**：在 API 测试阶段和数据库测试阶段之间插入 45s 冷却
2. **代码层面**：Python 请求封装指数退避重试（2s → 4s → 8s → 16s → 32s），尊重 `Retry-After` 响应头
3. **验证**：后续构建中 429 不再导致用例失败

**关键认知**：API 限流是服务端的保护机制，不是缺陷。测试设计需要主动考虑限流策略，不能依赖"碰运气"。

## 数据库一致性测试案例

### 场景：验证 API 创建的工作项在数据库中正确存储

**方法**：

1. 通过 API 创建工作项 → 获取返回的 ID 和字段值
2. 直接连接 Plane PostgreSQL 数据库 → 按 ID 查询对应行
3. 逐字段对比：名称、描述、状态、优先级、指派人等
4. 通过 API 更新工作项 → 再次查询数据库确认更新生效
5. 通过 API 删除工作项 → 确认数据库中对应行已删除

**价值**：这种交叉验证能发现 API 层面的"静默失败" — API 返回 200 但数据实际未正确持久化的场景。

## Flaky 用例判断方法

### 判断流程

```
用例失败
  ├─ retry → 通过
  │    ├─ 失败原因：超时 / 元素未找到
  │    │    └─ 分类：可能的 flaky（自动化稳定性）
  │    ├─ 失败原因：断言不匹配 / 业务逻辑错误
  │    │    └─ 分类：产品缺陷或测试脚本错误
  │    └─ 分析：手动复现是否稳定？
  │         ├─ 手动稳定 → flaky 确认
  │         └─ 手动也不稳定 → 产品缺陷
  └─ retry → 仍失败
       └─ 分类：产品缺陷或环境阻塞
```

### UI-007 应用此流程

1. 首次失败：超时 10s 未找到修改后的工作项
2. retry #1 → 通过
3. 失败原因是超时而非断言不匹配 → 倾向自动化稳定性
4. 手动复现稳定 → 确认为 flaky
5. 分类：自动化稳定性观察项，不修改代码

## AI 如何辅助但不替代人工判断

### AI 做了什么

- ChatGPT：生成测试方案、分析失败日志、输出报告框架
- Codex：编写 pytest/Playwright 代码、编写文档
- Claude Code：执行命令、部署服务、收集日志、拷贝文件

### 人工做了什么

- 审核所有 AI 生成的方案和代码
- 判断测试覆盖是否充分
- 决定是否修改 Jenkins 凭据
- 确认最终测试结论
- 区分 flaky vs 缺陷
- 确认所有报告内容的准确性

### 关键原则

> AI 加速执行和信息收集，但最终判断和责任由人工承担。

具体体现：
- AI 可以列出可能的原因（凭据 vs 代码 vs 服务），但选择从哪个环节开始排查是人的判断
- AI 可以生成测试代码，但断言覆盖是否充分需要人来审查
- AI 可以整理报告数据，但"建议交付"这个结论需要人来确认
